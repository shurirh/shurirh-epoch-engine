Los cambios afectan principalmente a:

Suceso

1. Cambios en el modelo Suceso
1.1 Restricción del tipo date

Actualmente:

date: number | string | Date | null

Esto genera ambigüedad y dificulta el razonamiento temporal.

Se cambia a:

date: number | null

Interpretación:

El número representa una unidad temporal abstracta definida por el usuario.

Ejemplos válidos:

-12000   // prehistoria
0        // punto de referencia
1492     // año histórico
2200     // futuro

El motor no impone unidades específicas.

1.2 Eliminación del campo branch

El campo:

branch: string | null

se elimina del modelo.

Motivo:

Las ramas temporales deben surgir de las relaciones del grafo, no de una propiedad del nodo.

Un mismo suceso puede pertenecer a múltiples ramas dependiendo de los caminos del grafo.

Si el usuario necesita registrar la rama de origen se recomienda usar meta.

Ejemplo:

meta: {
  originBranch: "timeline_alpha"
}
1.3 Protección de tags

Para evitar referencias externas mutables:

Antes:

this.tags = tags

Ahora:

this.tags = Array.isArray(tags) ? [...tags] : []

Esto asegura que la instancia de Suceso posea su propia copia.

1.4 Validación de meta

Asegurar que meta siempre sea un objeto plano.

this.meta = typeof meta === 'object' && meta !== null ? meta : {}
1.5 Modelo final de Suceso
Suceso
 ├─ id: string
 ├─ title: string
 ├─ description: string
 ├─ date: number | null
 ├─ tags: string[]
 └─ meta: object
2. Mejoras recomendadas en Graph

El grafo actual funciona correctamente pero se sugieren mejoras de eficiencia.

2.1 Índices de adyacencia

Actualmente getOutgoing y getIncoming recorren todas las aristas.

Esto escala mal en grafos grandes.

Se recomienda mantener índices internos:

outgoingIndex: Map<nodeId, Set<edgeId>>
incomingIndex: Map<nodeId, Set<edgeId>>

Esto permitirá:

getOutgoing(nodeId) -> O(k)
getIncoming(nodeId) -> O(k)

en lugar de:

O(E)
2.2 Validación de aristas duplicadas

Opcionalmente impedir duplicados lógicos:

A --causes--> B

no debería existir dos veces.

Esto puede validarse usando una clave:

from + type + to
3. Importación de datos (Feature futura)

Se planea incorporar un sistema para construir grafos a partir de datos externos.

Formato inicial sugerido:

JSON

Ejemplo:

{
  "events": [
    {
      "id": "e1",
      "title": "Descubrimiento de Marte",
      "date": 2032,
      "tags": ["space"],
      "meta": { "location": "Mars" }
    }
  ],
  "relations": [
    {
      "id": "r1",
      "from": "e1",
      "to": "e2",
      "type": "causes"
    }
  ]
}

El motor deberá poder:

importJSON(data)

Proceso:

1. Crear Sucesos
2. Insertarlos como Nodes en Graph
3. Crear Edges según relations
4. Validar con EpochEngine
4. Responsabilidades del sistema
Suceso

Representa información narrativa pura.

No contiene lógica temporal.

Graph

Administra:

nodos

relaciones

conectividad

No interpreta significado temporal.

EpochEngine

Responsable de:

validar causalidad

detectar paradojas

aplicar reglas narrativas

gestionar líneas temporales
